## Настройка среды разработки

```bash
# Клонирование репозитория
git clone https://github.com/hatter6822/seLe4n.git
cd seLe4n

# Установка тулчейна Lean (elan + Lean 4 + Lake)
./scripts/setup_lean_env.sh

# Сборка проекта
source ~/.elan/env && lake build

# Запуск трассировочного стенда
lake exe sele4n
```

### Установка pre-commit хука

Перед первым коммитом установите хук, блокирующий коммиты с ошибками
компиляции или `sorry`:

```bash
cp scripts/pre-commit-lean-build.sh .git/hooks/pre-commit
```

Хук автоматически:
1. Находит изменённые `.lean` файлы в staging-области
2. Собирает каждый модифицированный модуль через `lake build <Module.Path>`
3. Проверяет отсутствие `sorry` в staged-содержимом
4. **Блокирует коммит** при ошибке сборки или наличии sorry

Не обходите хук с помощью `--no-verify`.

## Обязательная валидация перед PR

```bash
./scripts/test_smoke.sh     # минимальный порог (Уровни 0-2)
./scripts/test_full.sh      # обязательно при изменении теорем/инвариантов (Уровни 0-3)
```

### Что проверяет каждый уровень

| Уровень | Скрипт | Содержание |
|---------|--------|------------|
| 0 | `test_fast.sh` | Гигиена кода: отсутствие `sorry`/`axiom`, целостность ссылок сайта |
| 1 | `test_fast.sh` | Компиляция: `lake build` завершается успешно |
| 2 | `test_smoke.sh` | Трассировка + негативные состояния + синхронизация документации |
| 3 | `test_full.sh` | Якоря поверхности инвариантов + проверка корректности Lean `#check` |
| 4 | `test_nightly.sh` | Ночной тест детерминизма (требуется `NIGHTLY_ENABLE_EXPERIMENTAL=1`) |

## Требования к Pull Request

### Чек-лист

- [ ] Указан идентификатор рабочего потока (workstream ID)
- [ ] Область изменений — один связный срез (coherent slice)
- [ ] Переходы (transitions) детерминированы — явный успех/ошибка
- [ ] Обновления инвариантов/теорем сопряжены с изменениями реализации
- [ ] `test_smoke.sh` проходит (минимум); `test_full.sh` для изменений теорем
- [ ] Документация синхронизирована в том же PR
- [ ] Не переименованы и не удалены пути, на которые ссылается сайт (см. `scripts/website_link_manifest.txt`)

### Структура коммитов

Придерживайтесь одного логического изменения на коммит. Сообщения коммитов
должны ясно описывать суть изменения.

### Правило Operations/Invariant

Каждая подсистема ядра разделена на:
- `Operations.lean` — переходы состояния (state transitions)
- `Invariant.lean` — доказательства сохранения инвариантов (preservation proofs)

Если вы изменяете переход, обновите соответствующие доказательства в том же PR.

## Ключевые правила кода

### Запрещённые конструкции

- **`sorry`** и **`axiom`** запрещены на продуктовой поверхности доказательств.
  Отслеживаемые исключения должны содержать аннотацию `TPI-D*`.
- **Недетерминированные ветвления** — все переходы возвращают явный результат
  успех/ошибка. Не вводите недетерминированные ветви.

### Типизированные идентификаторы

`ThreadId`, `ObjId`, `CPtr`, `Slot`, `DomainId` и другие — это структуры-обёртки
(wrapper structures), а не псевдонимы `Nat`. Используйте явные `.toNat`/`.ofNat`
для преобразований.

### Верификация модулей перед коммитом

**Перед коммитом любого `.lean` файла** необходимо убедиться, что конкретный
модуль компилируется:

```bash
source ~/.elan/env && lake build <Module.Path>
```

Например, при изменении `SeLe4n/Kernel/RobinHood/Bridge.lean`:

```bash
lake build SeLe4n.Kernel.RobinHood.Bridge
```

**`lake build` (цель по умолчанию) недостаточен.** Цель по умолчанию собирает
только модули, достижимые из `Main.lean` и тестовых исполняемых файлов.

## Синхронизация документации

При изменении поведения, теорем или статуса рабочих потоков обновите в том же PR:

1. `README.md` — синхронизация метрик из `docs/codebase_map.json`
2. `docs/spec/SELE4N_SPEC.md`
3. `docs/DEVELOPMENT.md`
4. Затронутые главы GitBook
5. `docs/CLAIM_EVIDENCE_INDEX.md` — при изменении утверждений (claims)
6. `docs/WORKSTREAM_HISTORY.md` — при изменении статуса рабочего потока
7. Регенерация `docs/codebase_map.json` при изменении исходников Lean

## Сообщение об уязвимостях

Если в ходе работы вы обнаружили потенциальную уязвимость, которая может
соответствовать критериям CVE, — **немедленно сообщите** о ней, не пытайтесь
исправить её молча. Это относится к:

- Логическим ошибкам в семантике переходов, проверках мандатов, контроле
  информационных потоков
- Известным или предполагаемым уязвимостям в зависимостях (Lean, Lake, elan)
- Небезопасным паттернам в скриптах сборки и CI
- Пробелам в формальной модели, создающим ложное чувство защищённости
