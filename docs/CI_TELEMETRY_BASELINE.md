# CI Telemetry Baseline (WS-B10)

This document is the canonical CI timing/flake telemetry baseline published by
**WS-B10 — CI maturity upgrades**.

## 1. Data sources

Telemetry is generated directly by repository-owned scripts:

- `scripts/ci_capture_timing.sh`
  - wraps CI commands and records duration + exit code JSON lines,
- `scripts/ci_flake_probe.sh`
  - repeats selected lanes and records attempt-level outcomes for flake signal.

Artifacts are uploaded from CI workflows to `.ci-artifacts/telemetry/`:

- Lean CI lanes (`.github/workflows/lean_action_ci.yml`): docs, fast, smoke, full,
- nightly lane (`.github/workflows/nightly_determinism.yml`): nightly + repeated smoke flake probe,
- ARM64 fast lane (`.github/workflows/platform_security_baseline.yml`).

## 2. Telemetry artifact schema

### 2.1 `timing.jsonl`

Each line is a JSON object:

```json
{
  "lane": "smoke",
  "command": "./scripts/test_smoke.sh",
  "start": "2026-02-18T06:01:10Z",
  "end": "2026-02-18T06:03:02Z",
  "duration_seconds": 112,
  "exit_code": 0
}
```

### 2.2 `flake_probe.jsonl`

Each line is a probe attempt object:

```json
{
  "probe_id": "nightly-smoke-repeat",
  "attempt": 2,
  "command": "./scripts/test_smoke.sh",
  "start": "2026-02-18T06:09:41Z",
  "end": "2026-02-18T06:11:35Z",
  "duration_seconds": 114,
  "exit_code": 0
}
```

### 2.3 `flake_summary.txt`

Human-readable summary generated by `scripts/ci_flake_probe.sh`, e.g.
`nightly-smoke-repeat: stable (3 attempts, exit=0)`.

## 3. Baseline interpretation policy

- A lane is considered **timing-regressed** when p95 duration grows >25% week-over-week
  for two consecutive weekly snapshots.
- A probe is considered a **flake signal** when repeated attempts yield mixed exit codes.
- Flake signals are triaged as reliability incidents and linked from the weekly CI review note.

## 4. Review cadence

- Weekly: review latest telemetry artifacts (scheduled workflows) and record drift notes.
- Per release: include notable timing/flake trends in `CHANGELOG.md` when materially relevant.

## 5. Scope note

This baseline is intentionally lightweight and artifact-driven to avoid introducing an external
telemetry backend as WS-B11 scenario telemetry requirements are now finalized.
